# Church Machine Security Model — Ring Definitions & Digital Security Reference

This document defines each enforcement ring in the Church Machine security model, explains the trust hierarchy, and addresses the full scope of digital-security questions that arise in a capability-based OS built on the Church Machine ISA.

---

## 1. Overview — The Dream Machine

The **Church Machine** is an ideal, flawless implementation of the lambda calculus Church Computer — a next-generation computer engineered for the everlasting Information Age. It demands stainless digital technology and eternal software reliability. Every component of the Church Machine, and the TSM (Trusted Security Machine) that enforces it, is designed to be minimal, verifiable, and perfectly transparent.

The Church Machine security model is **capability-based**: authority to perform an operation is conveyed by possessing the right **GT (Golden Token)** in a **CR (Capability Register)**. Successfully unlocked capability registers form the foundation of trust. There are no ACL tables, no ambient privilege, and no superuser. Every cross-boundary action is mediated by hardware or trusted microcode, and every capability is cryptographically anchored to the IDE at creation time.

**Crucial distinction**: All GTs are **local only**. Each thread has its own CR set in its call frame. A GT in one thread's CR is not visible to another thread, and the Namespace maintains only the gt\_seq revocation counter for each object\_id—not the GT values themselves. When the term "GT" appears in shared structures (like the Namespace), we refer to its identity (object\_id) and revocation state (gt\_seq), never its bit pattern. This locality is what enables true isolation: one thread's capabilities cannot leak to another.

**The core philosophy**: Anyone can code anything without impacting others, whatever mistakes they make. Errors in one lump do not corrupt another's capabilities or memory. This is the path to safe, everlasting software.

Six concentric enforcement rings are visualised in `TSM_SECURITY_RINGS.svg`. Ring 1 (centre) is most trusted; Ring 6 (outermost) is general software.

## 2. Ring Definitions

### Ring 1 — IDE (Cryptographic Origin)

| Property | Detail |
|---|---|
| Location | Off-device or privileged build-time tool |
| Trust basis | Private signing key, never present at runtime |
| Responsibility | Lump creation · CRC-16 generation · capability provenance |
| Runtime presence | **None** — IDE artefacts are verified, not executed |

The IDE is the root of all authority. It creates lumps, assigns object\_ids, embeds capability c-lists, and computes the CRC-16 over the GT Word 0 payload. Once a lump is signed and its CRCs are fixed, no runtime component can forge or extend those capabilities. All trust in the system ultimately reduces to "the IDE signed this lump with these capabilities."

The IDE never runs at runtime. Its influence is entirely static: the bytes it wrote, the CRCs it computed, and the capabilities it granted are the permanent record.

**Design commitment**: The IDE code itself is **minimal in size and open source for public review**. This transparency enables the community to audit and verify that no backdoors, trusts, or hidden assumptions are baked into the authority system.

**Ring 1 solves deep fakes**: Because every capability is cryptographically anchored at IDE signing time and verified through gt\_seq and CRC-16 at runtime, no impersonation or fraudulent capability can propagate through the system. The provenance of every GT is permanently traceable to the IDE that created it.

**Security properties delivered by Ring 1:**
- Capability provenance — every GT traces to a specific IDE-signed lump
- CRC-16 integrity — tampering with a GT Word 0 is detectable by ChurchNSGate
- Object\_id assignment — each class of object has a fixed, globally unique id
- PassKey generation — PassKey GTs (Abstract type, typ=11) are created here and never derivable at runtime
- Deep-fake immunity — no forged capability can pass gt\_seq and CRC-16 validation at runtime

### Ring 2 — ChurchNSGate (Shared Microcode Enforcement)

| Property | Detail |
|---|---|
| Location | Shared microcode layer, below all Church instructions |
| Trust basis | Hardware-enforced; no software-callable entry |
| Responsibility | NS slot lookup · gt\_seq revocation · CRC-16 verify · M-flag evaluation · GT typ dispatch |
| Runtime presence | Invoked by every CALL, RETURN, CHANGE, LOAD, SAVE |

ChurchNSGate is the universal enforcement checkpoint. Every Church instruction that crosses a lump boundary passes through ChurchNSGate before the operation is permitted. It performs, in order:

1. **NS slot lookup** — resolves the target CR to a Namespace slot; verifies the slot is populated and not revoked.
2. **gt\_seq check** — compares the gt\_seq field of the GT Word 0 against the Namespace's live counter; a stale gt\_seq means the capability has been revoked.
3. **CRC-16 verify** — recomputes CRC-16 over GT Word 0 bits [22:0] and compares to bits [38:23]; mismatch → fault.
4. **M-flag evaluation** — reads the volatile M flag from the hardware CR slot (not from GT Word 0); the M flag gates the LOAD CHECK\_L dual path.
5. **GT typ dispatch** — reads typ[24:23] and faults on NULL (00), routes Real (01), Outform (10), and Abstract (11) to their respective protocol paths.
   - **Outform GT (typ=10) resolution**: An Outform GT (O-GT) references an abstraction that is resolved dynamically by the Parent Namespace Manager (the parent IDE of this TSM). The O-GT's object\_id is looked up in the Parent IDE's registry, which returns an abstraction download (typically a zip distribution containing lump bytecode, dependencies, and metadata). Some abstractions are assigned as **resident** (pre-loaded in memory), while others are **cached** on demand, similar to web browser caching. The cached abstraction is fetched on first use, verified against the PIED (Parent IDE), and instantiated. This enables lightweight reference to large libraries and plugin ecosystems without embedding them.

ChurchNSGate cannot be bypassed, skipped, or called directly from software. Its invocation is implicit in every Church instruction that references a CR.

**Design commitment**: The ChurchNSGate microcode is **minimal in size**. There is only one thing to check, and we check it thoroughly: is this capability valid (gt\_seq + CRC-16), and if so, what type is it? Simplicity enables verification. The smaller and more focused the enforcement layer, the easier it is for the community to audit and validate that nothing is hidden.

**Security properties delivered by Ring 2:**
- Revocation — gt\_seq invalidates old copies of capabilities without hunting down every holder
- Integrity — CRC-16 detects GT Word 0 corruption or forgery
- Type safety — NULL GTs cannot be used; typ governs what operations are valid
- M-flag boundary — separates volatile hardware state (TSM-set) from software-visible GT fields
- Minimal attack surface — focused, small microcode means fewer lines to audit and fewer places an exploit can hide
- Abstraction caching — Outform GTs enable dynamic resolution of large libraries and plugin ecosystems via Parent IDE, with resident and cached (lazy-loaded) deployment models, reducing footprint without sacrificing capability

### Ring 3 — Church Instructions (Hardware ISA Boundary)

| Property | Detail |
|---|---|
| Instructions | CALL · RETURN · CHANGE · LOAD · SAVE · SWITCH |
| Location | ISA level — these are real opcodes in the 20-instruction Church Machine |
| Trust basis | Instruction decode and TSM microcode |
| Responsibility | Cross-lump control transfer · capability load/store · Namespace switching |

These six instructions are the only legal entry points into the capability enforcement machinery from software. They are **not** function calls; they are hardware instructions that trigger microcode sequences in the TSM.

| Instruction | Action |
|---|---|
| CALL | Transfer control to another lump's entry point; sets M flag on CR6 |
| RETURN | Return to the caller's lump; sets M flag on CR6 |
| CHANGE | Replace the running thread's lump context (tail-call form); sets M flag on CR6 |
| LOAD | Copy a GT from a source CR into a destination CR slot; dual-path via L perm or M flag; respects TPERM boundary |
| SAVE | Copy GT Word 0 from a source CR slot into a destination; M flag never propagated |
| SWITCH | Unlock a new Namespace application context; allows programmable security policies to be installed and enforced |

FORK and KILL are **absent** from this ring. They are not hardware instructions. Any lump attempting to fork or kill a thread must call into a Privileged Abstraction (Ring 5) and present a valid PassKey GT.

**The Church Machine Implements Programmable Security**: The programmer and the user decide the security policy, not the vendor. LOAD respects TPERM bits that gate whether a GT can be transferred. SWITCH enables different Namespace applications to manage their own lump hierarchies and enforcement rules. This is the opposite of fixed, vendor-controlled security.

**Example — Mint with FORK & KILL**: The Mint object (object\_id 0x0001) holds a PassKey GT (typ=Abstract, `0x01800100`) in its c-list. Only lumps that CALL into Mint and successfully present the PassKey GT can invoke Mint's FORK and KILL methods. When a lump calls Mint with a valid PassKey in the CR argument:
```
CALL Mint_entry, CR_with_passkey    ; ChurchNSGate validates PassKey GT
; Mint's FORK method runs, allocates a new thread
```
If the PassKey GT is revoked (its gt\_seq invalidated in the Namespace), the CALL into Mint fails at ChurchNSGate before Mint's code executes. Thread authority is entirely capability-based.

**Security properties delivered by Ring 3:**
- Controlled cross-lump transfer — CALL/RETURN/CHANGE pass through ChurchNSGate; arbitrary jumps are impossible
- Capability discipline — LOAD and SAVE are the only ways to move capabilities; no raw memory copy can replicate a GT because the M flag is hardware-volatile and CRC would be wrong
- Entry-point sealing — a lump can only be entered via its declared CALL target; no mid-lump entry
- Programmable policy — TPERM and SWITCH enable security rules to be decided at design time by the programmer, not at deployment time by the vendor
- Namespace applications — SWITCH allows new security contexts and lump hierarchies to be installed dynamically

### Ring 4 — TSM (Trusted Security Machine)

| Property | Detail |
|---|---|
| Location | Hardware microcode layer, inside the CPU |
| Trust basis | No software entry point; invoked only by Church instructions |
| Responsibility | M flag management · CR6 reload · Church instruction microcode |
| Runtime presence | Active on every CALL, RETURN, CHANGE |

The TSM (Trusted Security Machine) is the hardware microcode that implements the Church instructions. It is not a process, not a lump, and not callable from software. It has no ABI. Its only observable effect is:

- Setting the volatile **M flag** on the CR6 hardware slot when CALL, RETURN, or CHANGE is executed (indicating that CR6 was reloaded by a Church instruction, not by software)
- Clearing the M flag on any other CR6 reload
- Enforcing that LOAD's CHECK\_L path succeeds when either (a) the L permission bit is set in the source GT Word 0, or (b) the M flag is live on the source hardware slot

The M flag is **never** stored in GT Word 0. It is never included in the CRC. It is never copied by SAVE. It is hardware-volatile: it lives only in the CR slot, and only as long as the slot has not been overwritten by non-Church means.

**Security properties delivered by Ring 4:**
- M-flag boundary — TSM is the only entity that can set M=1; software cannot fake it
- Microcode integrity — CALL/RETURN/CHANGE semantics cannot be overridden by software
- CR6 authority — TSM controls when CR6 is considered "freshly loaded by the system"

### Ring 5 — Privileged Abstractions (PassKey-Validated Software)

| Property | Detail |
|---|---|
| Objects | Mint · Thread-manager · and any other object granted PassKey-issuing authority by the IDE |
| Location | Software layer — these are lumps, not microcode |
| Trust basis | PassKey GT (Abstract typ=11) validated by ChurchNSGate on entry |
| Operations gated here | FORK (create thread) · KILL (destroy thread) |

FORK and KILL are **methods** of privileged objects — primarily Mint — not hardware opcodes. To call FORK or KILL, the caller must present a PassKey GT in the appropriate CR. The PassKey GT is of Abstract type (GT Word 0 typ[24:23] = 11, `0x01800100`), issued by the IDE, and tied to a specific object\_id range (0x100–0x1FF by convention).

When Mint receives a CALL and the ChurchNSGate validates the PassKey GT, Mint is permitted to allocate a new thread or destroy an existing one. Without a valid PassKey GT in the CR, the CALL into Mint's FORK or KILL entry point will fault at ChurchNSGate before Mint's code ever runs.

This design means:
- FORK and KILL authority is **granted at compile time** by the IDE embedding a PassKey GT in the caller's c-list
- No runtime mechanism can create a PassKey GT — they can only be created by the IDE
- Revocation of FORK/KILL authority is immediate: invalidate the PassKey GT's gt\_seq in the Namespace and all holders lose access

**Security properties delivered by Ring 5:**
- Thread creation is gated — cannot FORK without IDE-granted PassKey capability
- Thread destruction is gated — cannot KILL without IDE-granted PassKey capability
- Privileged objects are still lumps — they are subject to the same CRC, gt\_seq, and typ checks as all other GTs
- No elevation — a general software lump cannot acquire a PassKey GT at runtime by any means

---

### Ring 6 — General Software / Lumps (Application Layer)

| Property | Detail |
|---|---|
| Location | Outermost ring — all application-level lumps |
| Trust basis | Only what capabilities the IDE embedded in the lump's c-list |
| Operations available | Turing instructions (DR arithmetic, branching) · Church instructions via held GTs |

General software lumps operate entirely on capabilities granted to them at build time. They cannot forge GTs, cannot set the M flag, cannot call TSM directly, and cannot acquire PassKey GTs at runtime. They are the clients of the security model, not its enforcers.

A lump in Ring 6 can still do significant work: it can CALL into other lumps (using Real GTs), LOAD capabilities granted to it (via L perm or M flag), and SAVE capabilities it holds. What it cannot do is exceed the authority the IDE chose to give it.

## 3. Digital Security — Frequently Asked Questions

### 3.1 How are capabilities forged or stolen?

They cannot be. A GT is a 64-bit value whose Word 0 bits [22:0] are covered by a CRC-16 stored in bits [38:23]. Any modification to the payload changes the CRC check result, and ChurchNSGate faults the instruction. A copy of a GT in raw memory (via Turing DREAD/DWRITE) is not a usable capability: to be used in a Church instruction, the GT must be in a CR slot and pass ChurchNSGate's CRC check. If an attacker copies GT bytes via DREAD/DWRITE into a CR slot, the CRC will pass — but the gt\_seq check will fail if the Namespace entry has been invalidated, and the M flag will be clear (it was not set by a Church instruction). So LOAD CHECK\_L will fault if L perm is not set on the GT.

### 3.2 How is a capability revoked?

By invalidating the gt\_seq counter in the Namespace. ChurchNSGate reads the gt\_seq field from GT Word 0 and compares it to the live Namespace counter for that slot. If they differ, the GT is stale and the instruction faults. Revocation is O(1) — the Namespace counter is incremented once and all outstanding copies of the old GT become inert simultaneously, without finding or modifying them.

### 3.3 What is the M flag and why can't software fake it?

The M flag is a hardware-volatile bit stored in the CR hardware slot register — not in GT Word 0, not in any software-visible memory. It is set to 1 exclusively by TSM microcode when a CALL, RETURN, or CHANGE completes and reloads CR6. Software has no write path to CR slot hardware flags; they are not memory-mapped. Therefore software cannot set M=1. The only entity that sets M=1 is the TSM, and that happens only when a Church instruction legitimately reloads the slot.

### 3.4 What is the PassKey GT and how is it created?

A PassKey GT is an Abstract GT (typ[24:23] = 11). Its canonical Word 0 pattern is `0x01800100` — the Abstract typ bits set in [24:23], plus the object\_id 0x100 occupying bits [7:0], with a valid CRC-16 in [38:23]. PassKey GTs are created exclusively by the IDE at lump compile time. There is no runtime API that issues a PassKey GT. A lump either has one in its c-list (put there by the IDE) or it does not.

### 3.5 What prevents a lump from calling FORK or KILL directly?

There are no FORK or KILL opcodes. They do not exist in the 20-instruction ISA. A thread-creation request is implemented as a CALL into Mint's entry point. If the caller does not hold a valid PassKey GT in the CR argument position, ChurchNSGate faults the CALL before Mint's code runs. Even if Mint's code is reached through some other path (impossible given the ISA), Mint internally verifies the PassKey on the received CR before allocating anything.

### 3.6 What prevents privilege escalation?

Several independent mechanisms in concert:

1. **No ambient privilege** — there is no ring-0 mode reachable by software; the TSM is microcode, not a callable context.
2. **No GT forgery** — CRC-16 prevents constructing a valid GT bit pattern without knowing the IDE's generation algorithm.
3. **No runtime PassKey issuance** — PassKey GTs cannot be created by any runtime code path.
4. **gt\_seq revocation** — even a legitimately obtained GT can be revoked by the Namespace owner.
5. **M flag isolation** — the M flag cannot be set by software, blocking one of the only ways to bypass LOAD's L perm check.
6. **Type gating** — ChurchNSGate dispatches on typ[24:23]; a Real GT cannot be used where an Abstract GT is required.

### 3.7 How does the system handle confused deputy attacks?

A confused deputy arises when a trusted component is tricked into using its authority on behalf of an untrusted caller. In the Church Machine, authority is conveyed by GTs in CRs, not by identity. When Mint receives a CALL, it receives exactly the CRs that the caller presented — no more. If the caller does not present a PassKey GT, Mint cannot use PassKey authority on the caller's behalf because it does not receive that GT in the call frame. There is no ambient authority that Mint holds independently of the call.

### 3.8 Can a lump inspect or enumerate another lump's capabilities?

No. CRs are per-thread and per-lump-context. A lump can only see the CRs in its own call frame. There is no introspection API that reads another lump's CR set. The Namespace stores live gt\_seq counters (not GT contents), so even the Namespace manager cannot read back capability payloads.

### 3.9 What does CRC-16 protect against?

The CRC-16 stored in GT Word 0 bits [38:23] covers the payload in bits [22:0]. It detects:

- Bit-flip errors (single or burst) in stored GTs
- Memory corruption that alters a GT's permission bits, object\_id, or type fields
- Naive forgery attempts that modify fields without recomputing the CRC

The CRC-16 does not protect against an attacker who knows the polynomial and can recompute it. The security guarantee against intentional forgery comes from the gt\_seq revocation and the M-flag isolation, not from the CRC being cryptographically strong. The CRC is an integrity check; the capability model's security against active adversaries comes from the hardware enforcement and the IDE's monopoly on PassKey issuance.

### 3.10 What happens if a lump's binary is tampered with after IDE signing?

The lump's embedded c-list (its GT constants) will have their CRCs checked by ChurchNSGate on first use. If a GT constant was altered, the CRC check fails and the instruction faults. Additionally, the gt\_seq values in tampered GT Word 0 fields will not match the live Namespace counters, causing a second independent fault path. A tampered lump cannot silently gain capabilities it was not given.

### 3.11 How is thread isolation enforced?

Each thread has its own Thread State Block (TSB) and its own CR set for the current call frame. Threads cannot read or write each other's CRs. Shared data must be mediated by a Real GT pointing to a shared memory region — and that GT must be legitimately held by both parties (placed in their c-lists by the IDE, or passed explicitly via a CALL). There is no shared address space by default.

### 3.12 What is the bootstrap security posture?

During bootstrap, sentinel GTs with reserved ids are used:
- `0xFFFF` — NS-table GT: grants access to the Namespace table itself; held only by the Loader
- `0xFFFE` — ROM Read GT: grants read access to ROM; held only by the Loader during init
- `0xFFFC` — NVFLASH GT: grants access to non-volatile storage; held only by the Loader

These sentinel GTs are never placed in general software c-lists. Once the bootstrap sequence completes and the Namespace is live, the sentinel GTs are expired (gt\_seq invalidated) and the live object\_id assignments (Loader=0x0000, Mint=0x0001, MM=0x0002, Locator=0x0003, Namespace=0x0004, Thread=0x0005, Tunnel=0x0006) govern all subsequent capability checks.

### 3.13 Can the Church Machine model express denial-of-service protection?

Partially. The capability model prevents unauthorised access but does not inherently throttle legitimate-capability-holding threads from consuming resources. DoS resistance must be built at the Privileged Abstractions layer: Mint can decline to FORK new threads when resource limits are reached, and Thread-manager objects can KILL runaway threads — but only if those objects were granted the relevant PassKey GTs by the IDE and are invoked by a suitably authorised lump.

## 4. Summary Table

| Ring | Name | Trust basis | Who enforces |
|---|---|---|---|
| 1 | IDE | Signing key | Off-device; static |
| 2 | ChurchNSGate | Hardware-enforced microcode | ChurchNSGate on every Church instruction |
| 3 | Church Instructions | ISA opcode decode | TSM microcode + ChurchNSGate |
| 4 | TSM | CPU microcode; no software entry | CPU hardware |
| 5 | Privileged Abstractions | PassKey GT (Abstract, IDE-issued) | ChurchNSGate + Mint/Thread-mgr logic |
| 6 | General Software | Only IDE-granted c-list capabilities | All outer rings collectively |
